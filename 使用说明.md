# P2P 聊天室使用说明

## 简介

这是一个支持 NAT 穿透的 P2P 聊天室应用。每个节点同时作为服务器和客户端运行，通过 WebSocket 进行通信。

### 功能特性

- 🌐 **双模式支持**：支持本地模式（localhost）和远程模式（跨局域网）
- 🔍 **自动端口发现**：自动在 1010-1100 端口范围内寻找可用端口
- 🌉 **NAT 穿透**：通过 STUN 服务器发现公网 IP，通过信令服务器进行对等发现
- 💬 **双界面**：支持控制台界面和 Web 界面
- 🔄 **全连接拓扑**：每个节点与所有其他节点建立直接连接

## 系统要求

- Go 1.25 或更高版本
- 支持的操作系统：Windows、Linux、macOS
- 网络要求：
  - 本地模式：无特殊要求
  - 远程模式：需要访问 STUN 服务器和信令服务器

## 安装

### 从源码编译

```bash
# 克隆仓库
git clone https://github.com/WetQuill/p2p-chatroom.git
cd p2p-chatroom

# 编译主程序
go build -o p2p-chatroom.exe .

# 编译信令服务器（远程模式需要）
cd server
go build -o signaling-server.exe .
cd ..
```

### 运行

```bash
# 直接运行（使用 Go）
go run main.go

# 或使用编译后的可执行文件
./p2p-chatroom.exe
```

## 使用模式

### 模式一：本地模式（默认）

本地模式适用于同一局域网或本机测试，无需额外配置。

#### 启动步骤

1. 启动第一个实例：
   ```bash
   go run main.go
   ```
   输出示例：
   ```
   成功接入:1010端口
   ```

2. 启动第二个实例（新终端）：
   ```bash
   go run main.go
   ```
   输出示例：
   ```
   成功接入:1020端口
   [系统] 用户 User-1 (ID:1) 已上线
   ```

3. 在任意实例输入消息，其他实例会收到：
   ```
   > 你好，我是用户1
   [User-2]: 你好，我是用户2
   ```

### 模式二：远程模式（跨局域网）

远程模式允许不同局域网的用户加入同一个聊天室。

#### 前提条件

1. 运行信令服务器（需要一个公网可访问的服务器）
2. 配置 `config.json` 文件

#### 步骤 1：启动信令服务器

```bash
cd server
go run main.go
# 或使用编译后的可执行文件
./signaling-server.exe
```

信令服务器将在端口 9000 上运行，输出：
```
Signaling server starting on :9000...
```

#### 步骤 2：配置客户端

复制配置模板：
```bash
cp config.json.example config.json
```

编辑 `config.json`：

```json
{
  "port": 0,
  "userName": "User-1",
  "mode": "remote",
  "signalingServer": "ws://your-signaling-server.com:9000/ws",
  "stunServers": [
    "stun:stun.l.google.com:19302",
    "stun:stun1.l.google.com:19302"
  ],
  "enableLocalMode": true
}
```

#### 配置说明

| 参数 | 类型 | 说明 | 默认值 |
|------|------|------|--------|
| `port` | int | 监听端口，0 表示自动发现 | 0 |
| `userName` | string | 用户显示名称 | 随机生成 |
| `mode` | string | 运行模式："local" 或 "remote" | "local" |
| `signalingServer` | string | 信令服务器地址（远程模式必需） | "" |
| `stunServers` | []string | STUN 服务器列表 | Google STUN 服务器 |
| `enableLocalMode` | bool | 是否启用本地模式连接 | true |

#### 步骤 3：启动客户端

```bash
# 使用默认配置文件
go run main.go

# 或指定配置文件路径
go run main.go -config path/to/config.json
```

成功启动后会显示：
```
[INFO] Configuration loaded:
[INFO]   Mode: remote
[INFO]   UserName: User-1
[INFO]   SignalingServer: ws://your-signaling-server.com:9000/ws
成功接入:1010端口
[INFO] Attempting STUN discovery...
[INFO] STUN discovery successful via stun:stun.l.google.com:19302: 203.0.113.1:54321
[INFO] Connected to signaling server: ws://your-signaling-server.com:9000/ws
[INFO] Received peer list with 2 peers
[系统] 已连接到 User-2 (ID:2)
```

## 控制台命令

在控制台界面中，支持以下命令：

### `exit`
优雅退出并向所有连接的对等节点发送下线通知。

```
> exit
```

### `checkconn`
显示当前连接状态，包括：
- 在线人数
- 每个对等节点的 ID、端口、用户名和远程地址

```
> checkconn

======= 当前连接情况 =======
当前在线人数: 2
ID      端口    用户名              远程地址
----    ----    ------              --------
1       1010    User-1              127.0.0.1:54321
2       1020    User-2              203.0.113.1:54322
===============================
```

## Web 界面

每个节点同时提供 Web 界面。

### 访问方式

在浏览器中打开：
```
http://localhost:节点端口/
```

例如，节点运行在 1010 端口：
```
http://localhost:1010/
```

### Web 界面功能

- 💬 发送和接收聊天消息
- 🔄 实时同步所有消息
- 🎨 简洁的用户界面

注意：Web 界面通过 WebSocket 与本地节点通信，消息会通过 P2P 网络广播给所有其他节点。

## 部署建议

### 信令服务器部署

生产环境建议：

1. **使用云服务器**：选择云服务提供商（AWS、阿里云、腾讯云等）
2. **配置防火墙**：开放 9000 端口（WebSocket）
3. **使用 HTTPS**：生产环境应配置 SSL/TLS
4. **负载均衡**：大规模部署时可考虑使用负载均衡器

示例使用 systemd 运行信令服务器：

```ini
# /etc/systemd/system/p2p-signaling.service
[Unit]
Description=P2P Signaling Server
After=network.target

[Service]
Type=simple
User=p2p
WorkingDirectory=/opt/p2p-chatroom/server
ExecStart=/opt/p2p-chatroom/server/signaling-server
Restart=always

[Install]
WantedBy=multi-user.target
```

### STUN 服务器

默认使用 Google 的公共 STUN 服务器。生产环境建议：
- 使用自建 STUN 服务器
- 或使用可靠的第三方 STUN 服务

常见公共 STUN 服务器：
```
stun:stun.l.google.com:19302
stun:stun1.l.google.com:19302
stun:stun2.l.google.com:19302
stun:stun.miwifi.com:3478
```

### 网络配置

#### 端口映射

如果客户端在 NAT 后面，需要配置端口转发：

1. **路由器端口转发**：将内网端口（如 1010）映射到公网端口
2. **UPnP**：如果路由器支持，可启用 UPnP 自动端口映射

#### 防火墙规则

确保防火墙允许：
- 入站：WebSocket 连接端口（1010-1100）
- 出站：到信令服务器的连接（9000）
- 出站：到 STUN 服务器的 UDP 连接

## 故障排除

### 问题：无法连接到信令服务器

**可能原因：**
- 信令服务器未运行
- 防火墙阻止连接
- URL 配置错误

**解决方法：**
1. 检查信令服务器是否运行：`curl ws://server:9000/ws`
2. 检查防火墙规则
3. 验证 `config.json` 中的 `signalingServer` 地址

### 问题：STUN 发现失败

**可能原因：**
- 网络不支持 UDP
- STUN 服务器不可达

**解决方法：**
1. 尝试不同的 STUN 服务器
2. 检查网络防火墙是否允许 UDP 出站
3. 联系网络管理员

### 问题：无法连接到对等节点

**可能原因：**
- NAT 类型不兼容
- 防火墙阻止
- 端口未映射

**解决方法：**
1. 启用 UPnP
2. 配置端口转发
3. 尝试直接连接（本地模式）
4. 未来版本将支持 TURN 服务器

### 问题：消息不显示

**可能原因：**
- Web 界面连接断开
- 节点未正确注册

**解决方法：**
1. 刷新浏览器页面
2. 检查控制台日志
3. 使用 `checkconn` 查看连接状态

## 技术架构

### 消息类型

| 类型 | 值 | 说明 |
|------|---|------|
| RegularMessage | 1 | 普通聊天消息 |
| JoinMessage | 2 | 节点上线 |
| ExitMessage | 3 | 节点下线 |
| JoinReplyMessage | 4 | 上线响应 |
| PeerDiscovery | 5 | 请求对等列表 |
| PeerList | 6 | 对等列表响应 |
| WebRTCOffer | 7 | WebRTC 提议（预留） |
| WebRTCAnswer | 8 | WebRTC 响应（预留） |
| ICECandidate | 9 | ICE 候选（预留） |

┌───┐ ┌───┐ ┌───┐ ┌───┐
